# Skill：保持文档与代码同步

## 核心原则（先看这条，再看下面）

只为 **「对外可见的事实」** 写文档。判定方法：

> 如果一个新成员（或下一次的 AI）**不读这个文件就无法正确使用项目**，那就是「对外可见」，必须进 README / .example.env。
> 否则属于实现细节，**不写文档**，必要时只在代码里加短注释。

这条原则 = 你不再需要背一长串清单，遇到任何新情况都能判断。

## 判定速查表

| 改动类型 | 是否进 README | 进哪段 |
| --- | --- | --- |
| 顶层目录新增 / 改名 / 删除（如根目录新建 `components/`、`docs/`） | ✅ 必须 | 目录结构与职责规范 |
| `app/**/page.tsx` 新增 / 删除 / 改路径 | ✅ 必须 | 页面与业务模块（路由表） |
| `app/api/**/route.ts` 路径 / 方法 / 入参 / 行为变化 | ✅ 必须 | HTTP 接口表 |
| `lib/config/sites.ts` 站点码 / env 前缀规则变化 | ✅ 必须 | 环境变量段 + 同步 `.example.env` |
| `package.json` 包管理器 / scripts / 依赖大版本变化 | ✅ 必须 | 技术栈 + 项目启动 |
| Shopify API 版本常量变化 | ✅ 必须 | Shopify API 版本来源 |
| `lib/` 下新增 / 改名 / 删除「业务模块文件」（如 `lib/shopify/products.ts`） | ✅ 必须 | 目录结构里 `lib/shopify/` 的注释文件列表 |
| 页面下 `_components/` 新增内部组件 | ❌ 不写 | 实现细节，README 不展开 |
| `lib/` 内函数签名、纯函数实现、内部重构 | ❌ 不写 | 必要时在该函数上方写一行注释 |
| 页面 UI 调整、文案变更、样式 | ❌ 不写 | — |
| 修 bug 不改对外行为 | ❌ 不写 | commit 信息说明即可 |
| 改测试 / 改注释 / 改 lint 配置但不影响开发命令 | ❌ 不写 | — |
| 新增「全仓库都要遵守」的约定 | ✅ 但不进 README | 进 `AGENTS.md` 或新建 `.cursor/rules/<topic>.mdc` |

> 边界情况：根目录新建一个 **空目录占位**、或一个临时实验文件——**先不写文档**，等真的成型有"对外用法"再写。避免文档抖动。

## 执行步骤

1. **盘点本次改动**：列出本轮新增 / 改名 / 删除 / 修改的文件路径。
2. **对照速查表**：逐项判定要不要进文档、进哪段。拿不准就回到「核心原则」自问一次。
3. **读再改**：用 `Read` 读目标章节，再用 `StrReplace` 最小化替换。**禁止整篇重写 `README.md`**。
4. **核对事实**：文档里出现的路径、命令、入参、env 名必须与当前代码一字不差。
5. **`.example.env` 同步**：仅当站点列表 / env 前缀规则变化时改动；保持站点条目 = `lib/config/sites.ts` 的 `CANDIDATE_SITE_CODES`。
6. **commit**：遵循仓库简体中文 + 类型前缀规范。文档单独提交用 `docs:`；与代码一起提交时沿用代码 commit 类型，正文里提一句"同步 README"。

## 自检（结束回合前必问）

- 本次改了哪些文件？每个文件对照速查表判定过了吗？
- README 里的目录树、路由表、API 表、env 段、版本来源段，有没有新事实没写进去？
- 文档里现存的路径 / 命令 / 变量名，是否都还能复制即用？
- 我有没有为了"显得在维护文档"而记录了实现细节（违反核心原则）？

通过这四问才算完成本次任务。

## 反例

- ❌ 改了一行 UI 文案就改 README。
- ❌ 在 README 里列出某页面所有内部组件文件名。
- ❌ 留 "TODO 之后补文档"。
- ❌ 把临时调试约定塞进 README，而不是 `AGENTS.md` / `.cursor/rules/`。
